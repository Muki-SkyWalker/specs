specs 1.5.0 API

This document is the API specification for specs 1.5.0 API

Class Summary
sealed abstract class CallConstraint extends AnyRef
This class is used to specify constraints on a sequence of expected calls.
It must defines a verifies method which says if the constraint is verified for a number n of consumed calls.
It must also define a stop method to specify if a SpecifiedCall can stop consuming received calls.
The expectation method returns a string describing what is expected by this constraint for example: "at least 3 of:". That string is used to form meaningful error messages
trait CalledInOrderMatchers extends ExpectableFactory with NumberOfTimes
This trait provides an additional matcher to check if some methods methods have been called in the right order on mocks: theMethod(m1.get(0)).on(m1).atLeastOnce then theMethod(m2.get(0)).on(m2) were called.inOrder // the implicit definition can also be removed but additional parenthesis are needed (m1.get(0) on m1).atLeastOnce then (m2.get(0) on m2) were called.inOrder
trait CalledMatchers extends CalledInOrderMatchers
This trait provides methods to declare expectations on mock calls: mockedList.get(0) was called mockedList.get(0) wasnt called mockedList.get(0) was notCalled where called is a Matcher which accepts supplementary methods to change the Mockito verification method: mockedList.get(0) was called.once // similar to was called mockedList.get(0) was called.twice mockedList.get(0) was called(3.times) mockedList.get(0) was called.atLeastOnce // or called.atLeast(oncee) mockedList.get(0) was called.atLeastTwice mockedList.get(0) was called.atLeast(3.times) mockedList.get(0) was called.atMostOnce mockedList.get(0) was called.atMostTwice mockedList.get(0) was called.atMost(3.times) mockedList.get(0) was called.exclusively // equivalent to called(0.timess)
trait ClassMocker extends Imposterizer
This trait allows to mock classes instead of interfaces only. This will require the cglib and objenesis libraries on the path.
class CountingNamingScheme extends org.jmock.api.MockObjectNamingScheme with AnyRef
Naming scheme adding a number to each anonymous mock
sealed case class Exclusivity (val isExclusive : Boolean) extends scala.Product
case class ExpectedCall (val method : java.lang.String) extends SpecifiedCall with scala.Product
Specifies that a method is expected to be called. By construction, the method name will include the line number in the original source file. The precondition is that 2 different expected methods should have 2 different names (so even in cases where a method is overloaded)
case class HamcrestMatcherAdapter [T](val m : Matcher[T]) extends org.hamcrest.TypeSafeMatcher[T] with scala.Product
Adapter class to use specs matchers as Hamcrest matchers
trait HamcrestMatchers extends AnyRef
This trait provide methods to build Hamcrest matchers. It actually contains only 2 methods since all Hamcrest matchers can be created by static methods in the Hamcrest library classes
trait HasInOrderVerificationMode extends Sugar
This trait provides functions to set the verification mode for InOrder verifications. This means that it only supports AtLeast and Times verification modes.
trait HasVerificationMode extends HasInOrderVerificationMode
This trait provides functions to set the verification mode
trait Imposterizer extends AnyRef
Abstract trait for creating a mocking context. By default, only allows to mock interfaces
trait InteractionMatchers extends ExpectableFactory
This trait provides an additional matcher to check if no more methods have been called on a mock: mock had noMoreCalls
trait JMockActions extends AnyRef
This trait provide methods to build jMock actions
trait JMocker extends JMockerExampleLifeCycle with HamcrestMatchers with JMockActions with ExpectationsListener
The JMocker trait is used to give access to the mocking functionalities of the JMock library
trait JMockerContext extends Imposterizer
This trait contains the jMock Mockery object and current expectations
trait JMockerExampleLifeCycle extends ExampleLifeCycle with JMockerContext
This trait defines when a jMock context will be created and expectations checked. The context and expectations are always created at the beginning of an Example, then checked just after the test has been executed. Executing a test can also trigger an ExpectationError (from the jMock library). This error will be transformed into a FailureException
trait Mocker extends ProtocolTypes with ExampleLifeCycle with MockMatchers with ExpectationsListener
This trait allows to define mocked method and to specify how they can be called It is used in 3 steps:
  • first create a mock object which will override the methods that you want to mock 
    val mock = new MyClass {
    override def method { record }
  }
  • implement the mocked methods with the Mocker.record method
  • declare expectations in your specification: expect { mock.method }
  • every expectation will be automatically checked at the end of the example (see the implementation of the ExampleLifeCycle trait)
trait Mockito extends MockitoLifeCycle with CalledMatchers with InteractionMatchers with MockitoStubs
The Mockito trait can be mixed with Specifications to provide mocking capabilities using the Mockito library. It provides some syntactic sugar on top of the Mockito methods and is integrated with specs expectations: object spec extends Specification with Mockito { val m = mock[java.util.List[String]] // a concrete class would be mocked with: mock(new java.util.LinkedList[String]) // stub a method call with a return value m.get(0) returns "one" // call the method m.get(0) // verify that the call happened, this is an expectation which will throw a FailureException if that is not the case m.get(0) was called m.get(1) wasnt called // we can also check that other calls did not occur } See the method descriptions for more usage examples.
trait MockitoLifeCycle extends ExampleLifeCycle
This trait allows the initialization of mocks when defined with an annotation: @Mock val l: List[String] = null The beforeTest method is overriden instead of the beforeExample method. This doesn't make a big difference but it helps the compiler in the Eclipse plugin not to crash too often.
trait MockitoStubs extends MocksCreation
This trait provides functionalities to declare stub values on method calls. Usage: mockedList.get(0) returns "one" mockedList.get(0) returns ("one", "two") mockedList.get(0) throws new Exception("unexpected") mockedList.get(0) answers ( i => "value " + i.toString ) It is also possible to chain stubs like this: mockedList.get(0) returns "one" thenReturns "two" mockedList.get(0) returns "one" thenThrows new Exception("unexpected now")
trait MocksCreation extends AnyRef
class Protocol extends ProtocolTypes
The Protocol class stores the expectations of mocks, alongside with the actual received calls It can be exclusive or not: i.e. return an error in case of unexpected calls or not
A protocol has 3 functions:
  • be defined by nested protocol definitions and expected calls
  • store received calls
  • return failures if there are unmatched calls
case class ProtocolDef (val protocolType : ProtocolType, val expectedCalls : scala.List[SpecifiedCall]) extends SpecifiedCall with ProtocolTypes with scala.Product
Specifies that a set of methods or other protocol definitions are expected to be called according to a protocol type
abstract class ProtocolType (repetition : CallConstraint) extends AnyRef
The ProtocolType specifies if a sequence of ReceivedCall can match a sequence of SpecifiedCall
trait ProtocolTypes extends AnyRef
This trait adds some frequent protocol types and some syntactic sugar to be able to specify such protocol types
  • anyOf = inAnyOrder(exactlyN(0))
  • oneOf = inAnyOrder(exactlyN(1))
  • 2.of = inAnyOrder(exactlyN(2))
  • 3.atLeastOf = inAnyOrder(exactlyN(3))
  • 3.inSequenceAtMostOf = inAnyOrder(exactlyN(3))
case class ReceivedCall (val method : java.lang.String) extends scala.Product
Specifies that a method has been received.
By construction, the method name will include the line number in the original source file.
abstract class SpecifiedCall extends AnyRef
Specifies which kind of received calls can be expected
case class atLeastN (val n : Int) extends CallConstraint with scala.Product
This class expects at least n received calls matching a specified call
case class atLeastNOf (val n : Int) extends inAnyOrder with scala.Product
case class atMostN (val n : Int) extends CallConstraint with scala.Product
This class expects at most n received calls matching a specified call
case class atMostNOf (val n : Int) extends inAnyOrder with scala.Product
case class exactlyN (val n : Int) extends CallConstraint with scala.Product
This class expects exactly n received calls matching a specified call
case class exactlyNOf (val n : Int) extends inAnyOrder with scala.Product
class inAnyOrder (val repetition : CallConstraint) extends ProtocolType
The inAnyOrder protocol type will try to consume expected calls in any order. It will not consume unexpected calls.
It accepts a repetition parameter specifying how many expected calls are supposed to happen:
  • exactlyN(2): exactly 2 times
  • atLeast(2): at least 2 times
  • atMost(2): at most 2 times
class inSequence (repetition : CallConstraint) extends ProtocolType
The inSequence protocol type will try to consume expected calls in sequence. It will not consume unexpected calls.
It accepts a repetition parameter specifying how many expected calls are supposed to happen:
  • exactlyN(2): exactly 2 times
  • atLeast(2): at least 2 times
  • atMost(2): at most 2 times
